# onco-pre-study

onco-pre-study is a pre-study diagnostics package for observational cancer research on OMOP CDM. It runs a configurable SQL analysis across a network of OMOP databases and produces a structured set of outputs documenting the empirical properties of a cancer cohort at each site: diagnosis coding patterns, metastasis timing, treatment records, co-occurring cancers, and mortality. It takes a data-driven approach to phenotype development: rather than inheriting assumptions from prior work, researchers can see how each database in the network represents the disease before defining inclusion and exclusion criteria and temporal logic. Cross-site comparison informs phenotype definitions that are robust across the network.

## Running

Open `run.R` and step through it block by block — do not `source()` it in one shot, as a failed query mid-loop will leave partial outputs with no clear indication of where it stopped.

**1. Configuration** — Set your CDM schema, target dialect, `min_cell_count`, and output directory. Fill in `DatabaseConnector::createConnectionDetails()` with your driver, server, port, and credentials.

**2. Connect** — `DatabaseConnector::connect()` opens the connection. An `on.exit()` call ensures it is closed even if a later step errors.

**3. Setup** — `DatabaseConnector::executeSql()` runs `00_setup.sql`, which builds all intermediate temp tables. This must complete successfully before any result chunk is run.

**4. Result loop** — For each numbered chunk (`01_` → `15_`), the helper calls `SqlRender::render()` to substitute `@cdm_database_schema` and `@min_cell_count`, then `SqlRender::translate()` to convert to the target dialect. `DatabaseConnector::querySql()` executes the translated SQL and returns a data frame, which is written to `outputs/<chunk_name>.csv`.

You can run the loop one iteration at a time if you want to inspect results as you go, or skip chunks that are not relevant.

## SQL dialect

The primary dialect is **SQL Server** (T-SQL) with [SqlRender](https://github.com/OHDSI/SqlRender)-style
parameters (`@cdm_database_schema`, `@min_cell_count`). Translations to 14 other dialects
are generated by `scripts/translate_sql_dialects.R`.

To render for a specific target database:

```r
library(SqlRender)
sql <- readSql("sql/sql_server/characterization_full.sql")
rendered <- render(sql, cdm_database_schema = "cdm", min_cell_count = 5)
translated <- translate(rendered, targetDialect = "postgresql")
```

## The characterization query

**Event families tracked:**


| Label | Source table         | Description                                 |
| ----- | -------------------- | ------------------------------------------- |
| DX    | condition_occurrence | Anchor UC diagnosis concepts                |
| ODX   | condition_occurrence | Other malignant neoplasm diagnoses          |
| GDX   | condition_occurrence | Generalized cancer ancestors of DX          |
| MET   | measurement          | Metastasis / AJCC stage 4 / M1 concepts     |
| L01   | drug_exposure        | L01 antineoplastic drugs (ingredient level) |


**Timing rules (both computed per run):**

- **FIRST** — earliest `event_date` per (anchor, family, concept, patient)
- **CLOSEST** — minimum `|days_diff|` to anchor date, tie-break by event_date

### Extended validation analyses (chunks 16–35)

A second batch of chunks, added for HUS validation, targets specific phenotype-design risks rather than general characterization. Each is tagged with a letter identifying which "Analysis" it belongs to (chunk filenames are `NN_<letter>_<topic>.sql`); the concept sets they depend on live alongside the main A–L setup sections in `00_setup.sql` (see section **E2** above). Analyses **D**, **G**, and **H** anchor on `#anchor_person` (has an anchor DX code, from any time) rather than `#cohort` (DX code additionally required to fall inside an observation period) — a broader population than chunks 01–15 use, so these analyses aren't blind to patients whose DX predates their earliest observation period.

- **Analysis E** (16–17) — characterizes the observation period itself: how much look-back/follow-up time patients actually have around each anchor, plus integrity checks (gaps, deaths outside periods, period type mix).
- **Analysis F** (18–19) — how repetitive and spaced-out the anchor DX/MET coding is: raw record counts per patient, and the day-scale between successive distinct DX codes.
- **Analysis D** (20–23) — a MET-first subgroup deep-dive: among DX-anchored patients, how often the metastasis code is recorded *before* the specific diagnosis that anchors the cohort, how well-supported that later DX is, and how long the lag is (bucketed and as a CDF).
- **Analysis H** (24–28) — the mirror question for treatment: where each patient's closest antineoplastic drug record sits relative to first metastasis, reconciling a CLOSEST-based view (histogram, before-curve) against a broader ANY-strictly-after view (after-curve), which deliberately use different denominators.
- **Analysis G** (29–32) — checks whether L01 drug_exposure alone under-captures treatment by also looking at Drug Therapy *procedure* codes: sizing procedure-only patients, which procedure concept drives the gap, its timing relative to MET, and how often procedures and drug_exposure records co-occur.
- **Analysis B** (33–35) — the coding trajectory of generalized/non-specific cancer diagnosis (GDX) codes around the first specific DX, to check whether an exclusion rule based on broad codes would wrongly drop patients whose broad code is really pre-diagnostic workup for the same disease.
